## 上下文

当前仓库的 WinDbg MCP 插件构建链路使用 CMake 输出产物，历史上存在文件名口径不统一的问题：配置、打包脚本与加载示例可能引用不同的 DLL 名称，导致发布和接入步骤存在误差。该变更围绕构建命名与文档一致性开展，不引入新的运行时协议或业务功能。

## 目标 / 非目标

**目标：**
- 统一主 DLL 产物名称为 `dbgx-mcp.dll`，并确保仓库内文档、示例和交付脚本使用同一命名口径。
- 建立可复用的验证点，避免将来再次出现同类命名漂移。

**非目标：**
- 不改变 MCP 协议能力、工具接口或 WinDbg 命令行为。
- 不新增依赖项、不引入新的构建系统以外的打包框架。

## 决策

- 决策 1：采用 CMake 原生产物命名（如 `OUTPUT_NAME`/`OUTPUT_NAME_DEBUG` 相关配置）统一 DLL 名称。
  - 替代方案：在构建后通过脚本重命名文件；该方案会增加 CI 与本地构建路径差异，且易引入时序问题，因此不采纳。
- 决策 2：在打包与加载示例文档中同步更新引用到新名称，并与 `README`/示例脚本复核。
  - 替代方案：仅更新构建配置不改文档，会导致使用者按旧名加载导致回归；不采纳。

## 风险 / 权衡

- [风险] 既有脚本/CI 缓存中仍引用旧文件名，导致发布流程失败。
  - 缓解：在提交前同步检索历史脚本和文档关键入口，并通过最小冒烟脚本验证 `dbgx-mcp.dll` 可加载。
- [风险] 一些外部自动化用户依赖旧文件名。
  - 缓解：在任务清单中加入兼容性提示与变更公告，必要时考虑后续补充兼容过渡策略。

## 迁移计划

- 按新名称重建产物。
- 更新所有 README、发布说明与示例中的 `.load` 路径。
- 验证构建与加载链路后再发布新版本。
