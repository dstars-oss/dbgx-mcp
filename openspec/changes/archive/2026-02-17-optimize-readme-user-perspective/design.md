## 上下文

当前 `README` 以实现与维护信息为主，首次阅读者需要先跨过构建细节才能理解项目价值与最小使用路径。虽然现有文档已覆盖 `.load` 示例与故障排查，但“为什么使用、怎么最快跑通”这条路径不够直接。

本次变更仅调整文档体验，不改变扩展 DLL、MCP 协议、构建脚本或运行时行为。主要约束是：命令示例必须可执行、双语文档结构需要保持一致、现有排障信息不能丢失。

## 目标 / 非目标

**目标：**
- 采用使用者优先的信息结构，让读者在前几屏明确项目价值、适用场景与能力边界。
- 提供可顺序执行的 Quick Start，覆盖“前置条件 -> 加载扩展 -> 首次 MCP 调用验证”。
- 保留并后置维护者向内容（测试策略、深入排障），减少首次使用的认知负担。
- 统一 `README.md` 与 `README.zh-CN.md` 的核心章节结构。

**非目标：**
- 不新增或修改 MCP 接口能力。
- 不改动 C++ 实现、CMake 配置、测试框架或导出符号。
- 不引入额外文档站点或自动化发布系统。

## 决策

1. 采用“用户旅程优先”章节顺序。
   - 决策：将 `README` 调整为“项目概览 -> 适用场景 -> 快速开始 -> 常见问题/排障 -> 深入信息”。
   - 原因：该顺序直接对应使用者首次试用流程，能缩短首次成功路径。
   - 备选方案：保留原顺序仅补充目录。未采用，因为目录无法解决信息优先级问题。

2. 保留原有技术内容，但下沉到后置章节。
   - 决策：构建细节、测试策略、完整排障仍保留，不在首屏展开。
   - 原因：避免信息丢失，同时降低首次阅读噪声。
   - 备选方案：删除维护者内容。未采用，因为会削弱项目可维护性与诊断能力。

3. Quick Start 使用“最小可验证闭环”而非分散命令示例。
   - 决策：将前置条件、`.load`、`initialize`/`tools/call` 串成连续步骤，并给出预期结果提示。
   - 原因：使用者按步骤执行即可判断成功与否，减少试错。
   - 备选方案：维持分散的示例片段。未采用，因为读者难以判断执行顺序和完成标准。

4. 双语文档采用“结构对齐、内容等价”原则。
   - 决策：中英文 README 采用一致的核心章节与顺序，命令示例保持等价。
   - 原因：避免不同语言用户获得不一致的上手体验。
   - 备选方案：仅保证主文档更新，翻译后补。未采用，因为会造成长期漂移。

## 风险 / 权衡

- [风险] 文档总长度增加，可能影响快速浏览。  
  [缓解] 通过清晰目录与章节命名，让“快速开始”入口前置。

- [风险] 双语文档更新不同步。  
  [缓解] 在任务中加入同步与结构一致性检查，合并前双文件对照审阅。

- [风险] 命令示例随构建流程演进而过时。  
  [缓解] 在文档更新时执行一次最小命令验证，并保留可追踪的测试/排障入口。

## 迁移计划

1. 重写 `README.md` 的章节结构与首屏信息。
2. 同步调整 `README.zh-CN.md`，保证章节对齐与命令等价。
3. 执行一次文档自检：按 Quick Start 逐步核对命令可执行性与预期结果。
4. 保留既有排障与测试章节，确认未丢失关键诊断信息。

回滚策略：若合并后反馈可读性下降，可回退至上一版 README，并保留本变更中的 Quick Start 段落作为最小增量。
