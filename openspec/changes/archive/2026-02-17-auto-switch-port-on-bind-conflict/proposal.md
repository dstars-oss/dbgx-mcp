## 为什么

当前 MCP HTTP 服务依赖固定端口；当端口已被其他进程占用时，扩展启动会失败并需要人工改端口后重试，影响可用性与自动化场景稳定性。需要让服务在端口冲突时具备自恢复能力，以降低启动失败率并减少手工排障。

## 变更内容

- 在 MCP HTTP 服务启动阶段增加端口冲突恢复：若端口绑定失败（已占用），自动尝试后续端口直到找到可用端口或达到尝试上限。
- 定义并约束端口选择策略（起始端口、递增规则、最大尝试次数、失败返回条件）。
- 在 WinDbg 可见输出中记录端口冲突与最终绑定端口，便于客户端连接与问题定位。
- 保持现有 JSON-RPC 协议与工具语义不变；本变更不引入 BREAKING 行为。

## 功能 (Capabilities)

### 新增功能

- 无

### 修改功能

- `windbg-http-mcp-eval`: 扩展 HTTP 端点启动需求，要求在监听端口冲突时自动切换到可用端口，并对最终监听端口提供可见反馈。

## 影响

- 受影响代码：HTTP 监听初始化与端口配置逻辑、启动日志输出逻辑。
- 受影响测试：需新增/更新端口占用与回退场景测试，覆盖成功回退与回退失败路径。
- 外部影响：客户端不再能假设始终绑定单一固定端口，需要读取启动输出或约定的端口发现方式。
